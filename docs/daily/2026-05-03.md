## Архитектурные решения и почему

- **Путь A: стратегия отдаёт LetterResponse целиком.** Не Путь B (только incident-часть). Причина: для багажных кейсов (Day 4, Montreal Convention) и subject, и body будут принципиально другими — лучше изоляция чем общий шаблон с условностями.
- **`BODY_TEMPLATE` дублируется в DelayLetterStrategy и CancellationLetterStrategy.** Намеренно — для будущих стратегий он будет другим. Преждевременная DRY-абстракция тут вредна.
- **Validation осталась в orchestrator (`ClaimLetterServiceImpl.validate`).** Стратегия предполагает что claim уже валидный.
- **Добавил проверку `issue.getType() != null`** — раньше эта проверка была неявной через `switch`; в делегат-варианте нужна явно, иначе `strategiesByType.get(null)` упадёт с менее полезным сообщением.

## Замеченные нюансы

## Что было поправлено по дороге (silent bugs)

При первой реализации обнаружились три проблемы — поправили:

1. **Не было `@Component`** на обеих новых стратегиях — Spring не подхватывал их в инжекшен. Тесты проходили, потому что не было ни одного теста на новый тип. На API-вызове с `MISSED_CONNECTION` упало бы `IllegalArgumentException("No eligibility strategy for type MISSED_CONNECTION")`. Это типичный silent bug — компилируется, тесты зелёные, но фича в проде сломана. Закрыли тестами на новый тип, теперь регрессия словит.

2. **Группировка скобок в `inScope`** была неправильной:
   ```java
   // было — null pointer на Boolean.getDepartureFromEu() через ||
   Boolean.TRUE.equals(euContext.getDepartureFromEu() || Boolean.TRUE.equals(euContext.getEuCarrier()))
   // стало
   Boolean.TRUE.equals(euContext.getDepartureFromEu()) || Boolean.TRUE.equals(euContext.getEuCarrier())
   ```

3. **Опечатка "filght"** в incident line — была скопирована из DelayLetterStrategy. Поправили во всех трёх стратегиях.

## Архитектурные решения и почему

- **Реиспользовали `Issue.delayMinutes`** для итоговой задержки прибытия после missed connection. Не добавляли отдельные поля типа `totalArrivalDelayMinutes` или `connectingFlightNumber`. Семантически — одно и то же значение «как поздно пассажир прибыл», просто причина разная.
- **Required documents такие же** что у DELAY/CANCELLATION (TICKET + BOARDING_PASS). По правилам EU 261 для missed connection нужны boarding pass обоих сегментов, но в текущей модели один document type, поэтому пока упрощение.

## Замеченные нюансы

- Архитектура стратегий **сработала как задумывалось**: `EligibilityServiceImpl` и `ClaimLetterServiceImpl` не тронуты, Spring сам подхватил новые `@Component`-классы. Day 3 — первая практическая проверка паттерна.
- **Silent bug с забытым `@Component`** показал важность тестов на каждый новый тип. Без них регрессия не словила бы — тесты прошли бы, и фича сломалась бы только в проде. Урок: для каждого нового strategy-bean писать минимум один интеграционный тест, который проходит через весь pipeline.
- **Дубликат `BODY_TEMPLATE` в трёх стратегиях** уже немного зудит. Если для Day 4 (BAGGAGE_DELAYED) шаблон таки будет принципиально другим — оставляем. Если окажется что и его можно подогнать под общий — пора рефакторить в shared template.

## Что было на плане

- Letter-стратегия с принципиально другим subject и телом (Montreal, не EU 261).

## Что реально сделано

1. `IssueType.BAGGAGE_DELAYED` добавлен в enum.
2. **Первая миграция Flyway V2** — `V2__add_baggage_delay_fields.sql`:
   ```sql
   alter table issues
       add column baggage_delay_hours integer;
   ```
3. `Issue` сущность + `IssueRequest` DTO расширены полем `baggageDelayHours` (без `@Column` — Hibernate авто-конвертит camelCase в snake_case).
4. `BaggageDelayedEligibilityStrategy`:
   - `MIN_DELAY_HOURS = 6`, `PER_DAY_AMOUNT = 50`, `MAX_AMOUNT = 500`.
   - Eligibility: не extraordinary + `baggageDelayHours >= 6`.
   - Compensation: `Math.ceil(hours / 24.0) * 50`, capped на 500 EUR.
   - Required documents: `PIR + BAG_TAG`.
   - **EuContext не используется** — Montreal Convention применяется к международным рейсам без географической привязки к EU.
5. `BaggageDelayedLetterStrategy`:
   - Subject — `"Baggage Delay Compensation Claim - <flight> (<route>)"`, без EU 261.
   - Body — упоминает Montreal Convention 1999 Article 19, обращение к Baggage Services Team.
6. **Тесты:**
   - 4 юнит-теста в `EligibilityServiceImplTest`: eligible на 24ч, не eligible на 4ч, extraordinary, capped на 40 днях.
   - Интеграционный тест в `ClaimIntegrationTest` с хелпером `buildBaggageDelayedIssue(int)`.
   - `ClaimServiceImplTest.setUp()` обновлён — обе baggage-стратегии в списках.
7. Все тесты зелёные.
8. Коммит — `feat(claim): add BAGGAGE_DELAYED claim type (Montreal Convention)`.

## Что было поправлено по дороге (silent bugs)

При первой реализации обнаружились критичные проблемы — поправили:

1. **Неправильное имя таблицы в миграции** — `alter table issue` вместо `alter table issues`. Flyway упал бы на старте с `relation "issue" does not exist`.

2. **`BaggageDelayedLetterStrategy` была копией `DelayLetterStrategy` без адаптации:**
   - Subject содержал «EU 261» вместо «Baggage Delay Compensation Claim».
   - Использовался `getDelayMinutes()` вместо `getBaggageDelayHours()` (для багажа `delayMinutes` обычно null).
   - Тело шаблона содержало литерал `<hours>` вместо плейсхолдера `%s` — клиент получил бы письмо с этой строкой как есть.
   - Переменная `incidentLine` создавалась, но не использовалась в `formatted(...)`.
   - Отступы в шаблоне разной длины — лесенка в письме.

3. **Избыточная проверка `issue.getType() == BAGGAGE_DELAYED`** в стратегии — orchestrator уже маршрутизирует по типу.

Все эти ошибки были пойманы LLM-ревью до коммита. Урок: для каждой новой стратегии особенно важно проверять, что не остался copy-paste из соседней — ошибки тут тихие, тесты могут не словить (нужны тесты на конкретный тип).

## Архитектурные решения и почему

- **Новое поле `baggageDelayHours`, не реиспользуем `delayMinutes`.** Семантика принципиально другая: задержка багажа в часах/днях vs задержка прибытия рейса в минутах. Реиспользование привело бы к `delayMinutes = 4320` для «3 суток» — loaded gun.
- **Без `@Column` на новом поле.** Hibernate с `SpringPhysicalNamingStrategy` сам преобразует `baggageDelayHours` в `baggage_delay_hours`. Существующие поля `delayMinutes`/`cancellationNoticeDays` тоже без аннотации — соблюдаем единообразие.
- **PIR mandatory через `requiredDocuments`, не через отдельную булеву колонку.** Используем существующий `DocumentTypes.PIR`. Чище и не плодит схему.
- **Per-day allowance, не actual reasonable expenses.** Montreal Convention требует возмещения по чекам, но для MVP фиксированная сумма по дням — приемлемое упрощение, близкое к реальной практике авиакомпаний для мелких сумм.
- **Без SDR/EUR конверсии.** Лимит зашит в EUR. Реальная конверсия через ЦБ/fixer.io — отдельная фича.

## Замеченные нюансы

- **Флоу `EligibilityResult.requiredDocuments` расширяется естественно.** Для baggage просто другой `List.of(...)` — не пришлось менять интерфейс или сервис.
- **Архитектурное решение Day 2 (полный body в каждой стратегии, без shared template) сегодня окупилось.** Body для baggage реально другой — другое обращение, другая правовая основа. Если бы мы вынесли «общий шаблон» в Day 2, сейчас пришлось бы рефакторить с if'ами.
- **Migration в TestContainers** — Flyway применяется в интеграционных тестах автоматически, никаких настроек не требуется.
